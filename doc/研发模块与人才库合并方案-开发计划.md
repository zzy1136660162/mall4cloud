# 研发模块与人才库合并方案开发计划

## 1. 目标与业务关系

本次改造基于已经创建的独立 uni-app 工程 `front-end/mall4cloud-rd-uniapp`，从原商城小程序 `front-end/mall4cloud-uniapp` 中迁移研发模块和人才库模块，合并成一个独立的研发小程序。

前端继续使用 uni-app。后端和管理端继续复用现有 Mall4Cloud 服务，必要时只在 `mall4cloud-platform` 中补充研发小程序免登录接口和研发需求字段，不新建后端服务，不新建管理后台。

第一期按“免登录访问、低门槛留资”设计：用户可以直接浏览研发门户、查看专家与成果库、提交研发需求。需求提交通过联系人、联系电话、需求编号承接，不把登录作为首个转化门槛。

业务关系一句话：

```text
研发门户聚合平台能力和专家资源，引导企业提交研发需求；
需求表单沉淀产品品类、期望服务、对接领域；
后台根据需求字段反查 talent_pool 专家能力标签，完成线下匹配和项目转化。
```

第一期核心闭环：

```text
研发门户
  -> 8 环节流程展示：讲清楚平台能把项目推进到哪里
  -> 4 大业务模块展示：讲清楚平台具备哪些核心能力
  -> 研发需求表单：收集企业需求和匹配线索
  -> 专家与成果库入口：展示资源组织能力
  -> 免登录需求查询：用手机号 + 需求编号查看进度
```

## 3. 关键产品设计

### 3.1 研发门户定位

研发门户不是普通营销首页，也不是原“研发需求中心”的简单改名。它应该承担三个任务：

- 讲清楚平台的产业转化路径。
- 让企业快速提交研发需求。
- 让专家与成果库成为可信资源背书。

建议首页由 4 个核心区块组成：

1. 8 环节流程
   - 资源入库
   - 需求征集
   - 项目立项
   - 概念验证
   - CDMO 转化
   - IP 运营
   - 科学传播
   - 市场转化

2. 4 大业务模块
   - 高校专家与成果协同
   - 概念验证与 CDMO 协同
   - 知识产权运营
   - 专业应用评议与科学传播

3. 研发需求 CTA
   - 主按钮：立即提交研发需求
   - 次入口：需求进度查询
   - 第一版免登录，不展示“我的全部需求”，避免账号体系依赖

4. 专家与成果库入口
   - 展示推荐专家或能力方向
   - 跳转 `talent-pool/index`
   - 强化“资源组织能力”，不是单纯人才黄页

### 3.2 研发需求与人才库的关联字段

任务书里提出的 3 个字段建议保留，它们是研发模块和人才库真正关联起来的关键：

| 字段 | 数据库列 | 作用 |
| --- | --- | --- |
| `productCategory` | `product_category` | 区分体表健康产品、功能性食品 |
| `serviceType` | `service_type` | 记录企业希望获得的服务包 |
| `expertiseField` | `expertise_field` | 对接人才库的专家能力领域 |

业务逻辑：

```text
企业提交需求
  -> 选择 productCategory / serviceType / expertiseField
  -> 需求落库 rd_demand
  -> 管理端查看需求
  -> 根据 expertiseField 在 talent_pool.expertise_areas 中匹配专家
  -> 线下对接、项目立项、概念验证或转化服务
```

这 3 个字段建议第一期就做，原因是它们不只是表单项，而是研发需求和人才库合并后的业务连接点。

## 6. 后端改造建议

### 6.1 研发需求字段扩展

建议扩展 `rd_demand`，新增 3 个字段：

```sql
ALTER TABLE `rd_demand`
    ADD COLUMN `product_category` tinyint DEFAULT NULL
        COMMENT '产品品类: 1-体表健康产品 2-功能性食品'
        AFTER `functional_appeal`,
    ADD COLUMN `service_type` varchar(200) DEFAULT NULL
        COMMENT '期望服务类型(JSON数组)'
        AFTER `product_category`,
    ADD COLUMN `expertise_field` varchar(100) DEFAULT NULL
        COMMENT '期望对接领域'
        AFTER `service_type`;
```

同步改造：

- `Demand.java`
- `DemandSubmitDTO.java`
- `DemandVO.java`
- `DemandMapper.xml`
- `DemandServiceImpl.submit()`

字段写入规则：

- `productCategory` 存数字枚举。
- `serviceType` 存 JSON 数组字符串，例如 `["1","3"]`。
- `expertiseField` 存与人才库领域一致的中文枚举。

### 6.2 免登录提交接口

现有 `POST /mall4cloud_platform/demand/submit` 会读取 `AuthUserContext`，不适合免登录研发小程序。

建议新增：

```text
POST /mall4cloud_platform/ua/demand/submit
```

请求体兼容 `DemandSubmitDTO`，并支持新增 3 个字段：

```json
{
  "title": "舒缓修护精华研发",
  "functionalAppeal": "希望验证舒缓修护功效，关注敏感肌场景",
  "productCategory": 1,
  "serviceType": "[\"1\",\"2\",\"3\"]",
  "expertiseField": "功效评测",
  "targetAudience": "敏感肌用户",
  "dosageFormPreference": "精华液",
  "budgetRange": "100000-200000",
  "expectedDeliveryTime": "2026-09-30",
  "remark": "希望同步评估中试可行性",
  "submitterName": "张先生",
  "submitterPhone": "13800000000"
}
```

接口要求：

- 不读取登录用户上下文。
- `submitterName`、`submitterPhone` 必填。
- 后端生成 `demandNo`。
- 默认状态为待处理。
- 返回需求 ID 和需求编号，方便前端展示查询凭证。

建议响应：

```json
{
  "id": 12345,
  "demandNo": "RD202606220001"
}
```

### 6.3 免登录查询接口

建议新增：

```text
GET /mall4cloud_platform/ua/demand/query?demandNo=&submitterPhone=
GET /mall4cloud_platform/ua/demand/detail/{id}?demandNo=&submitterPhone=
```

安全要求：

- 必须同时校验 `demandNo` 和 `submitterPhone`。
- 不允许只按手机号列出全部需求。
- 第一版不开放撤回、重新申请、删除。

## 7. 前端表单方案

研发需求表单字段：

| 字段 | 文案 | 必填 | 说明 |
| --- | --- | --- | --- |
| `title` | 项目/产品名称 | 是 | 如舒缓修护精华、控糖功能饮品 |
| `functionalAppeal` | 研发目标与功能诉求 | 是 | 功效、成分、技术难点、验证目标 |
| `productCategory` | 产品品类 | 是 | 体表健康产品、功能性食品 |
| `serviceType` | 期望服务类型 | 是 | 可多选，专家匹配、概念验证、CDMO 等 |
| `expertiseField` | 期望对接领域 | 否 | 与人才库能力标签一致 |
| `targetAudience` | 目标用户/应用场景 | 是 | 人群、场景、渠道 |
| `dosageFormPreference` | 产品形态偏好 | 否 | 精华、面膜、饮品、软糖等 |
| `budgetRange` | 预算范围 | 是 | 沿用预算选项 |
| `expectedDeliveryTime` | 期望交付时间 | 是 | 日期选择 |
| `remark` | 补充说明 | 否 | 其他说明 |
| `submitterName` | 联系人 | 是 | 免登录核心留资 |
| `submitterPhone` | 联系电话 | 是 | 免登录核心留资和查询凭证 |

提交成功页或弹窗必须展示：

- 提交成功。
- 需求编号。
- 联系电话。
- 1-3 个工作日内联系的说明。
- 保存需求编号用于后续查询。

## 8. 人才库合并方案

“人才库”在研发小程序中改为“专家与成果库”或“研发专家库”，第一期继续复用 `talent_pool`。

列表页调整：

- 搜索文案：搜索专家、方向或成果。
- 筛选标签优先使用 `EXPERTISE_FIELD_OPTIONS`。
- 如果后端不支持领域筛选，第一期只做前端展示或本地筛选，不承诺服务端筛选。

详情页调整：

- 基本信息：姓名、头衔、地区、年限、学历。
- 专注领域：`expertiseAreas`。
- 专业能力：`skills`。
- 项目经验：`projectExperience`。
- 成果荣誉：`achievements`。
- 文案强调“专家能力、成果基础、可协同方向”。

## 9. 分阶段实施计划

### 阶段一：新工程基线整理

目标：让 `front-end/mall4cloud-rd-uniapp` 成为干净可开发的研发小程序工程。

任务：

- 忽略或删除 `unpackage/` 生成产物。
- 确认使用 HBuilderX 构建，还是补齐 `package.json` 后用命令行构建。

- 手工维护 `pages.json`，不要复制原商城 `pages.json`。

- 专家库列表和详情正常。
- 不出现登录弹窗或 token 过期拦截。

## 10. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 原 uni-app 页面存在编码或模板损坏 | 编译失败 | 迁移时重写关键页面，不直接复制破损文件 |
| 免登录接口扩大数据暴露 | 隐私风险 | 查询必须同时校验手机号和需求编号 |
| 新增字段与人才库标签不一致 | 无法匹配专家 | 统一使用 `utils/dict.js` 字典 |
| 后端不返回 `demandNo` | 用户无法查询 | `/ua/demand/submit` 必须返回需求编号 |
| 新工程没有 `package.json` | 命令行验证困难 | 明确 HBuilderX 构建或补齐依赖管理 |
| 管理端未展示新增字段 | 后台无法匹配专家 | 管理端沿用原页面时至少确保详情接口返回字段 |

## 11. 后续增强

- 项目协同工作台。
- 8 环节节点状态跟踪。
- 专家智能匹配。

- 服务包商品化：专家匹配包、概念验证包、转化落地包、IP 运营包、传播赋能包。
