# 研发小程序双人并行开发任务书

## 1. 项目目标

基于 `front-end/mall4cloud-rd-uniapp` 开发独立研发小程序，将研发申请模块和人才库模块合并成一个“研发门户”。

第一期产品原则：

- 小程序端免登录。
- 用户可直接浏览研发门户、查看专家与成果库、提交研发申请表单。
- 用户提交研发申请后，数据落到 `rd_demand`。
- 商家端/平台端通过需求管理能力查看申请，并线下联系客户。
- 人才库作为平台研发资源和专家能力展示，不做用户线上选专家交易。
- 两个静态页面/区块展示公司研发能力，第一期不做后台配置。

核心业务关系：

```text
用户进入研发小程序
  -> 浏览研发门户和公司研发能力
  -> 查看专家与成果库
  -> 填写研发申请表单
  -> 需求落库 rd_demand
  -> 商家端/平台端查看需求
  -> 线下联系客户并匹配 talent_pool 专家资源
```

## 3. 分工原则

本次不按“前端/后端”拆分，而是按业务板块拆分。每位同学都要负责自己板块的前端、后端、接口联调和自测。

| 同学 | 负责板块 | 前端范围 | 后端范围 |

| A 同学 | 研发门户 + 研发申请板块 | UI 设计、首页、公司研发能力展示、研发申请表单、提交成功页 | `rd_demand` 新字段、免登录提交接口 |
| B 同学 | 专家与成果库 + 需求查询 + 商家端流转板块 | 专家库列表/详情、需求查询/详情 | 人才库接口保障、免登录查询接口、商家端/平台端流转验证 |

特别要求：

- Stitch 设计由 A 同学负责。
- A 同学生成 Stitch 设计稿后，必须先交给 PM 看效果，PM 确认后再进入页面开发。
- B 同学可以在 A 等待 UI 确认时先做后端查询接口和人才库接口验证。
- 两位同学都可以使用 Trae AI 辅助，但接口字段、路径、业务逻辑必须人工确认。

## 8. 代码规范

### 8.1 小程序前端规范

- 使用 uni-app + Vue 3。
- 页面文件放在 `pages/`。
- 研发门户组件放在 `components/rd/`。
- 公共接口放在 `utils/api/`。
- 字典放在 `utils/dict.js`。
- 不写登录判断。
- 不携带 token 请求公开接口。
- 表单必须做必填校验。
- 接口字段使用 camelCase。
- 页面文案统一使用中文业务术语。
- 样式参考 A 的 Stitch 设计稿。

### 8.2 后端规范

- 不引入 Lombok，沿用项目 getter/setter 风格。
- Controller 返回 `ServerResponseEntity.success(data)`。
- 公开接口放在 `DemandUaController`。
- MyBatis XML 字段、VO 字段、DTO 字段必须同步。
- 新增 SQL 只追加 ALTER，不破坏原 CREATE。
- 免登录查询必须同时校验手机号和需求编号。
- 不开放免登录删除、撤回、重提接口。

### 8.3 Trae AI 使用规范

允许：

- 生成 Vue 页面骨架。
- 生成 CSS/SCSS 样式。
- 生成 Java getter/setter。
- 生成 SQL 模板。

必须人工确认：

- 接口路径。
- 字段名。
- 是否免登录。
- 是否写入 `rd_demand`。
- 查询是否校验手机号和需求编号。

禁止：

- 让 AI 自行发明接口。
- 让 AI 自行改动对方负责的文件。
- 复制 AI 生成的登录逻辑。
- 复制 AI 生成的支付、订单、商城相关代码。
