# Mall4Cloud 微服务精简部署规划

## 1. 当前服务作用速览

| 服务 | 端口 | 作用 | 是否在交易主链路 | 精简判断 |
| --- | ---: | --- | --- | --- |
| `mall4cloud-gateway` | 8000 | 统一入口、路由到各业务服务、聚合 Knife4j 文档 | 是 | 保留。除非改前端直连各服务，否则不能停。 |
| `mall4cloud-auth` | 9101 | 登录、Token 签发/刷新/退出、验证码、社交登录 | 是 | 保留。登录态存在 Redis。 |
| `mall4cloud-rbac` | 9102 | 权限、菜单、角色、用户角色关系，鉴权过滤器会调用 | 是 | 保留。小系统可长期考虑并入 auth。 |
| `mall4cloud-leaf` | 9100 | 分布式 ID 生成，订单、用户、店铺、平台需求等会调用 | 是 | 保留。短期停掉风险高。 |
| `mall4cloud-user` | 9105 | 会员、地址、用户资料、达人/人才相关用户能力 | 是 | 保留。 |
| `mall4cloud-product` | 9104 | 商品、分类、品牌、SKU、购物车、库存锁定 | 是 | 保留。 |
| `mall4cloud-order` | 9106 | 下单、订单详情、取消、收货、订单状态统计 | 是 | 保留。 |
| `mall4cloud-payment` | 9113 | 支付单、支付回调、通知订单支付成功 | 有支付时是 | 如果暂时不在线支付，可停；否则保留。 |
| `mall4cloud-biz` | 9109 | OSS/MinIO 上传、附件、短信等通用业务能力 | 取决于上传 | 小程序上传图片、后台上传素材时需要；纯静态资源可弱化。 |
| `mall4cloud-multishop` | 9103 | 店铺、商家、首页轮播/热搜、店铺信息 | 取决于店铺功能 | 当前小程序会调用店铺详情、轮播、热搜、开店接口；不用店铺可裁。 |
| `mall4cloud-platform` | 9112 | 平台后台、系统配置、需求/人才库等平台业务 | 取决于当前定制功能 | 当前小程序需求、人才库接口在这里；保留或拆出轻量模块。 |
| `mall4cloud-search` | 9108 | 商品搜索、商品管理搜索、订单搜索，依赖 ES 索引 | 否，可替代 | 第一优先级降级对象。可改 MySQL 直查后停用。 |
| `mall4cloud-api` | - | Feign 内部接口定义 | 编译依赖 | 不是运行服务。 |
| `mall4cloud-common` | - | 公共能力：缓存、数据库、安全、RocketMQ、订单公共类 | 编译依赖 | 不是运行服务。 |

## 2. 当前性能压力来源

当前 `scripts/services.conf` 里 12 个 Java 服务的 JVM `Xmx` 合计约 **7GB**：

- 10 个服务各 `512m`：约 5GB。
- `ORDER` 和 `SEARCH` 各 `1024m`：约 2GB。

中间件还会额外占用内存：

- Elasticsearch：配置 `-Xms512m -Xmx512m`，实际还会吃文件缓存和 native 内存。
- RocketMQ namesrv + broker：各约 512m。
- Nacos：约 256m。
- MySQL、Redis、Seata、Canal、MinIO 也会持续占用。

所以单台 4G 或 8G 服务器全量启动会很吃力，尤其是 **Search + ES + Canal + RocketMQ** 这条同步链路。优先方向不是直接合并所有服务，而是先停掉高成本、非核心链路。

## 3. 推荐精简路线

### 阶段一：先降配置，不改业务

目标是快速降低内存峰值，适合当天处理服务器扛不住的问题。

- 保留核心服务：`gateway`、`auth`、`rbac`、`leaf`、`user`、`product`、`order`、`payment`、`biz`。
- 只有在小程序仍使用店铺、轮播、热搜、开店申请时保留 `multishop`。
- 只有在小程序仍使用需求提交、人才库、平台配置时保留 `platform`。
- 如果暂时不使用商品搜索页、店铺商品列表、订单搜索列表，不启动 `search`、`elasticsearch`、`canal`。
- 关闭非必要管理后台容器，例如 RocketMQ dashboard。
- 将非核心 Java 服务的 `Xmx` 从 `512m` 下调到 `256m`，`order/product` 保持 `512m` 起步，观察 GC 和接口延迟。

这个阶段不解决搜索功能依赖，只是先保命。若前端仍访问 `/mall4cloud_search/**`，停 search 会导致页面报错。

### 阶段二：把 search 从 ES 改成 MySQL 直查

这是最建议做的精简点。当前搜索服务覆盖：

- 小程序商品列表：`/mall4cloud_search/ua/search/simple_page`。
- 小程序搜索列表、店铺商品列表、店铺首页商品：同样走 `simple_page`。
- 平台商品管理：`/mall4cloud_search/p/search/page`。
- 商家商品管理：`/mall4cloud_search/m/search/page`。
- 小程序订单列表：先走 `mall4cloud-order`，再 Feign 调 `SearchOrderFeignClient` 到 search。
- 商家订单列表：同样依赖 search。

建议替代方案：

- 商品搜索 v1：在 `mall4cloud-product` 增加 MySQL 分页查询，支持关键词、分类、店铺、状态、价格、库存、排序等常用条件。
- 商品聚合 v1：先不做 ES 那种品牌/分类/属性聚合，或只返回简单品牌/分类列表。小程序优先保证商品列表可用。
- 商品管理页：平台/商家后台列表直接使用已有 `SpuService.page/platformPage` 思路，补齐当前搜索页需要的条件。
- 订单搜索：在 `mall4cloud-order` 内部 MySQL 查询 `order`、`order_item`、`order_addr`，支持订单号、商品名、收货人、手机号、状态、时间、用户/店铺过滤。
- 保持接口返回结构尽量兼容，先让前端少改或不改，后续再清理 `mall4cloud-api-search`。

完成后可以停用：

- `mall4cloud-search`
- `mall4cloud-elasticsearch`
- `mall4cloud-canal`
- 商品/订单同步 ES 的 RocketMQ 消费链路

收益最大：少一个 Java 服务、少 ES、少 Canal，并减少 MQ 同步压力。

代价：

- MySQL 模糊搜索不如 ES，关键词分词、相关性排序、属性聚合会变弱。
- 数据量上来后要补索引，例如商品名、分类、店铺、状态、创建/更新时间、订单状态、用户、店铺、订单创建时间。
- 如果商品量超过数十万或搜索体验要求高，长期仍应保留 ES；但当前服务器扛不住时，直查更现实。

### 阶段三：评估 RocketMQ 和 Seata

RocketMQ 现在不是只服务 search，它还参与：

- 下单后的库存锁定/解锁。
- 订单取消延迟处理。
- 支付成功后通知订单。
- Canal 到 search 的索引同步。

因此不要第一步直接停 RocketMQ。建议：

- search 替换完成后，去掉 Canal 同步 Topic，RocketMQ 压力会下降。
- 如果订单量很低，可以把“订单超时取消、库存解锁、支付通知”改为数据库状态 + 定时任务补偿，再考虑停 RocketMQ。
- 支付通知不建议只靠同步 HTTP，至少要有补偿任务，避免支付成功但订单未更新。

Seata 当前用于跨服务事务，用户、店铺、订单、商品、平台等有 `@GlobalTransactional`。短期建议保留。只有当核心链路改成单服务内本地事务，或跨服务操作都有补偿机制后，才考虑移除。

## 4. 按业务场景裁剪服务

### 场景 A：完整商城最小可用

保留：

- `gateway`
- `auth`
- `rbac`
- `leaf`
- `user`
- `product`
- `order`
- `payment`
- `biz`
- `multishop`
- `platform`

可替换后停用：

- `search`
- `elasticsearch`
- `canal`

中间件保留：

- MySQL
- Redis
- Nacos
- Seata
- RocketMQ
- MinIO

适合仍需要商品、购物车、下单、支付、店铺、需求提交、人才库的小程序。

### 场景 B：轻商城，不做店铺/商家体系

保留：

- `gateway`
- `auth`
- `rbac`
- `leaf`
- `user`
- `product`
- `order`
- `payment`
- `biz`
- `platform`（如果需求/人才库还在用）

可裁剪：

- `multishop`：前提是移除店铺详情、开店申请、店铺首页、店铺商品等入口。
- `search`、ES、Canal：改 MySQL 直查后裁剪。

风险：

- 商品详情页当前会请求店铺头部信息。
- 搜索结果会补店铺信息。
- 首页旧版有轮播图从 multishop 获取。

因此要先确认小程序是否还展示店铺概念。

### 场景 C：只做展示/需求提交，不做交易

保留：

- `gateway`
- `auth`（如果需要登录）
- `rbac`（如果后台权限还用）
- `leaf`
- `user`
- `platform`
- `biz`（如果有图片上传）

可裁剪：

- `product`
- `order`
- `payment`
- `search`
- `multishop`
- ES、Canal、RocketMQ

适合只做首页、人才库、需求提交、资料上传的轻量业务。前端需要隐藏购物车、商品、订单、支付、店铺入口。

## 5. 服务合并建议

不建议一上来把所有 Spring Boot 服务合成一个单体。这个成本高、容易引入循环依赖，短期也不一定比“停掉不用的服务 + 降低 JVM”更省。

更稳妥的合并顺序：

1. `search` 并入 `product/order`：先用 MySQL 直查替代 ES。
2. `auth + rbac`：登录鉴权和权限查询强相关，可作为中期合并对象。
3. `biz + platform`：如果 biz 只剩上传，platform 又承载定制业务，可以合并通用后台能力。
4. `order + payment`：支付只服务订单时可合并，但要谨慎处理支付回调、幂等和补偿。
5. `leaf`：最后处理。可以改为数据库号段、本地雪花 ID 或 MySQL 自增，但会影响很多表和 `@DistributedId` 逻辑。

优先级最高的是第 1 项，因为它直接减少 ES、Canal 和一个 Java 服务。

## 6. 推荐执行顺序

1. 记录当前服务器内存、CPU、磁盘 IO、各 Java 进程 RSS、MySQL 慢查询。
2. 先把 `services.conf` 中非核心服务 JVM 下调，停 dashboard 等管理容器。
3. 梳理小程序实际菜单，确认哪些入口必须保留：商品、订单、支付、店铺、需求、人才库。
4. 做 search 替换方案：商品列表和订单列表先 MySQL 直查，接口返回尽量兼容。
5. 灰度验证 search 替换：商品列表、搜索页、店铺商品、订单列表、平台/商家商品列表。
6. 停 `mall4cloud-search`、ES、Canal，观察 1-3 天。
7. 再评估是否要把 RocketMQ 的订单延迟能力改为数据库定时补偿。
8. 最后才考虑服务合并。

## 7. 我建议的目标架构

短期目标，适合小服务器：

```text
Nginx/小程序
    -> gateway
        -> auth/rbac
        -> user
        -> product  （商品列表 MySQL 直查）
        -> order    （订单列表 MySQL 直查）
        -> payment
        -> platform （需求/人才库）
        -> biz      （上传）
        -> multishop（仍需要店铺时保留）

中间件：
MySQL + Redis + Nacos + Seata + RocketMQ + MinIO

停用：
search + Elasticsearch + Canal
```

如果后续确认不做在线支付、不开店铺、不做商城交易，可以继续裁掉 `payment`、`multishop`、`product`、`order` 和 RocketMQ。

## 8. 关键风险和验收点

- 停 search 前，所有 `/mall4cloud_search/**` 调用必须有替代接口。
- 订单列表不能只按订单表查，搜索商品名时需要关联 `order_item`。
- 商品列表直查要注意库存、上下架状态、分类、店铺分类、价格排序、销量排序。
- MySQL 直查上线前必须补必要索引，否则 ES 停掉后压力会转移到 MySQL。
- 停 Canal 后，商品/订单不会再同步 ES；如果以后恢复 ES，需要做全量重建索引方案。
- 停 RocketMQ 前，必须有订单超时取消、库存释放、支付成功补偿方案。
- 降低 JVM 后要观察 GC、接口耗时和 OOM 日志，不要只看进程是否启动成功。

## 9. 结论

当前最现实的精简路线是：

1. 先降 JVM 和停非必要管理容器。
2. 把 `mall4cloud-search` 的商品/订单查询改为 MySQL 直查。
3. 停 `search + Elasticsearch + Canal`。
4. 保留 RocketMQ 和 Seata，等交易链路稳定后再做更深裁剪。

这样能在不大规模重构的前提下，先砍掉最重的一组服务，同时保住小程序商品、订单、支付等核心功能。
